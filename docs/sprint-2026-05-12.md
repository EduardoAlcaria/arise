## 1. Live deployment logs (GitHub Actions style) 🔥

**Problem:** Logs are batched and only readable after deployment finishes. `DeploymentWatcher` opens an SSE stream but the implementation polls DB every 1s and creates a new unbounded thread pool per request — it leaks threads and delivers logs in chunks, not as they happen.

**Goal:** Each `appendLog()` call inside `executeAsync` / `executeApplicationDeploy` fans out immediately to any connected client — no polling, no wait.

### Backend changes

**Replace DB-poll SSE with a pub/sub in-memory fan-out:**

- `LogBroadcaster` — singleton Spring bean holding a `ConcurrentHashMap<Long, Set<SseEmitter>>` (deploymentId → connected emitters)
- `DeploymentController.streamLogs()` — registers a new `SseEmitter`, adds it to the broadcaster, replays existing DB logs on connect so the client catches up if it joins mid-run, then keeps the connection open
- `DeploymentService.appendLog()` — after saving to DB, calls `logBroadcaster.publish(deploymentId, message)` which pushes to all live emitters for that deployment
- On emitter timeout/error → remove from broadcaster map
- Use a shared `ScheduledExecutorService` (fixed pool, not `newSingleThreadExecutor` per request)

**Files to touch:**
- Create: `service/LogBroadcaster.java`
- Modify: `service/DeploymentService.java` — inject LogBroadcaster, call publish in appendLog
- Modify: `controller/DeploymentController.java` — rewrite streamLogs to use broadcaster + replay

### Frontend changes

`DeploymentWatcher.tsx` already uses EventSource — no structural change needed. Logs will now arrive line-by-line in real time. Style the log panel like GitHub Actions: monospace, dark bg, auto-scroll to bottom, timestamp prefix per line.

## 2. `.env` injection in single-repo deploys 🔥

**Problem:** `DeployRepoWizard` collects env vars from `.env.example` in step 2, shows them in step 3, but `buildEnvFileConfigItem()` result is **never passed** for single-repo mode — only for application mode. Any real app deploys without config.

**Files to touch:**
- `frontend/src/components/DeployRepoWizard.tsx` — in `handleDeploy` single-mode branch, include `envFile` in the `DeployItem`
- `frontend/src/api/deployments.ts` — add optional `envVars: Record<string, string>` or `envFile: string` to `DeployItem` / `DeploymentRequest`
- `backend/.../dto/deployment/DeploymentRequest.java` — add `configFiles` field (same as application mode)
- `backend/.../service/DeploymentService.java` — after git clone, if `configFiles` present, write them into `repoDir` before running build command

## 3. Redeploy button

**Problem:** No way to push an update. Users create a new deployment each time which pollutes history and breaks the "runs" grouping.

**Implementation:**
- `POST /api/deployments/{id}/redeploy` — clones the same repo + branch again and reruns build/compose. Reuse all fields from the original deployment, create a new `Deployment` row linked to same `repositoryUrl` (so it groups with existing runs).
- Frontend: add a Redeploy button (↺ icon) in the expanded runs list next to the Rollback button — visible on `SUCCESS` and `FAILED` runs.

## 4. Real rollback

**Problem:** `rollback()` in `DeploymentService` sets `status = ROLLED_BACK` and nothing else. The running containers are untouched.

**Implementation:**
- On rollback, SSH into the machine and run `docker compose down` in the current deploy dir, then `docker compose up -d` in the previous deploy dir (store `deployDir` on the `Deployment` entity).
- If the deploy was a single-repo stack-detection deploy: store the `repoDir` path (`/tmp/deploy_{id}`) on the entity. Currently ephemeral — needs to be persisted so rollback can re-enter it.
- Short-term acceptable: rollback runs a new deploy from the same repo/branch — effectively a re-deploy of the last good commit SHA (store `resolvedCommitSha` on entity after clone).

## 5. Infrastructure topology + animations 🗺️

**Goal:** A visual animated graph of everything Arise knows, extended with read-only AWS data and Terraform state. No EC2 connection or SSM — pure visibility.

---

### Phase A — Our own infra (build first, 1 session)

Data already available in Arise:
- Machines (host, status, tunnel type)
- Deployments (machine → repo → stack → Cloudflare tunnel URL)
- Containers (machine → image → port)
- Cloudflare tunnels (hostname → deployment)

**Backend — `GET /api/topology`**

Returns a typed graph ready for the frontend to render:
```json
{
  "nodes": [
    { "id": "machine-1", "type": "machine", "label": "Mac Mini", "status": "ONLINE", "meta": {...} },
    { "id": "deploy-4",  "type": "deployment", "label": "retrospective-ai", "status": "SUCCESS", "meta": {...} },
    { "id": "tunnel-1",  "type": "tunnel", "label": "retrospectiveai.alcaria.dev", "meta": {...} }
  ],
  "edges": [
    { "source": "deploy-4", "target": "machine-1", "label": "deployed on" },
    { "source": "tunnel-1", "target": "deploy-4",  "label": "exposes" }
  ]
}
```

**Frontend — `/topology` page**

Library: **React Flow** (`@xyflow/react`) — fits the tech stack, no extra dependencies.

- Nodes styled as mini-cards matching the existing Arise dark theme
- Color by health: accent green = online/success, destructive red = error/failed, muted = unknown/pending
- Animated edges for active data paths (React Flow's `animated` edge prop)
- Click node → slide-in side panel with details + quick actions (test SSH, open terminal, view logs, open tunnel URL)
- Auto-layout with dagre (`@dagrejs/dagre`) so the graph arranges itself

**Animations:**
- Deploying: pulsing border on the deployment node + animated edge from machine to deployment
- ONLINE machine: subtle green glow
- Tunnel active: animated dashed edge from tunnel → deployment → machine (shows the request path)
- Failed: red flash on node, broken edge style

### Phase B — AWS read-only layer

**Goal:** Pull AWS resource data and overlay it on the same graph. Read-only — no actions, no SSM.

**Backend:**
- `AwsCredentialsService` — store AWS access key + secret + region per user (Settings page)
- Validate on save via `sts:GetCallerIdentity`
- `AwsReadService` — calls AWS SDK v2 (read-only):
  - `ec2:DescribeInstances` → EC2 nodes
  - `ec2:DescribeVpcs` + `DescribeSubnets` → grouping/container nodes
  - `ec2:DescribeSecurityGroups` → edge annotations
  - `elasticloadbalancing:DescribeLoadBalancers` → LB nodes
  - `rds:DescribeDBInstances` → DB nodes
  - `ecs:ListClusters` + `ListServices` → ECS service nodes
- `GET /api/topology` extended: when AWS creds configured, merges AWS nodes/edges into the response
- AWS nodes linked to Arise machine nodes where IPs/instance IDs match

**Frontend:**
- AWS nodes rendered with a different icon set (EC2, RDS, ALB icons from lucide or custom SVGs)
- VPCs rendered as background grouping rectangles behind their member EC2/RDS nodes
- Toggle button: "Show AWS layer" / "Hide AWS layer"
- AWS credentials section in Settings (same pattern as Cloudflare)

---

### Phase C — Terraform state parsing

**Goal:** Upload or point to a `terraform.tfstate` and see it overlaid on the graph. Drift detection against live AWS.

**Backend:**
- `POST /api/terraform/state` — accepts raw tfstate JSON (multipart upload)
- Parser extracts: resource type, name, attributes, `depends_on` chains → topology nodes + edges
- `GET /api/terraform/drift` — for each Terraform resource that has a cloud counterpart, compare attribute values to live AWS describe call → flag `drifted: true` on node
- Store parsed state in DB (one `TerraformState` entity per user, replaceable)

**Frontend:**
- Settings: upload `.tfstate` file or paste S3 path
- Terraform layer toggle in topology view
- Drifted nodes shown with amber border + warning icon
- Hovering a drifted node shows diff: "Terraform says X, AWS says Y"

### Phase D — Animated request/deploy flow

The "payload flow" from the planned features — animate how a request travels:

```
Browser → Cloudflare tunnel → machine → container → database
```

And during a deployment:

```
GitHub → Arise → SSH → machine → docker compose
```

Each step lights up sequentially as it happens, tied to real deployment log events. This uses the same React Flow graph with timed edge animations driven by SSE log events.
